# LESSON-008 · Beta 期不解决"未来痛点"

## 触发场景

2026-04-26 ADR-0072 起草过程:

1. Stage 5 发现 Practice 三套 KP ID 格式并存(4 位数字 / ikp.* / board-prefixed)
2. Claude 第一反应:推荐选 C 双指针架构 + 立 ADR-0074 拆 L0 字段
3. NZH 让综合分析 · Claude 重审约束:
   - Solo dev + Beta 期 + 无活功能因三格式 break
   - 5.5d 迁移估算 → 实际 8d+
   - 选 C 隐藏陷阱:双重维护 → 不可靠的统一字段 → 架构债更深
4. 转向建议:**推迟 ADR-0072 到 Exit-Beta(M6+)** · 改 ADR-0012 1 行措辞消除冲突(无需立新 ADR)
5. NZH 同意 · 节省 8d 工程时间

## 问题

**Claude 默认倾向"先架构后验证"** · 表现为:
- 看到 schema 不一致 → 立刻提议统一架构
- 看到 ADR 措辞冲突 → 立刻提议立新 ADR
- 看到三套并存 → 立刻提议双指针 / 中间层
- 估算时低估迁移成本(乐观偏差 1.5-2x)

**Beta solo dev 真实约束**:
- 每个工程日都是 学生体验 / 灵魂功能 的机会成本
- "未来可能用上"的架构在 Beta 期 99% 不会被消费
- ADR-0064 change governance:**架构 commitment 是不可逆的(第 4 级)** · 应延迟决策直到必须

## 规则

**Why**:Beta 期 25Maths < 50 学生 + 无 SLA + 无商业化。任何"为 M3 / M5 准备"的架构 = 给未来一个可能错的自己写代码。AI 默认乐观估算 + 默认完美主义 → 必须显式反向校准。

**How to apply**:
1. **任何架构提议先回答**:
   - 当前 prod 因此 break 吗?(若否 → 推迟)
   - Beta 期 6 个月内有活功能消费这个架构吗?(若否 → 推迟)
   - 不做这事的真实痛在哪?(若答不出 → 推迟)
2. **优先选 1 行措辞修订 over 新 ADR**(参见 ADR-0012 vs AP-002 案例)
3. **优先选 opt-in 字段 over 强制迁移**(参见 TASK-022 修订)
4. **估算迁移成本 × 1.5x** 作为 baseline(乐观偏差校正)
5. **决策推迟 ≠ 决策回避**:Deferred-to-MN 是合法 ADR 状态 · 写明重启条件即可

## 反例(2026-04-26 第一版)

```
Stage 5 发现三格式并存
  → Claude 立 ADR-0072 仲裁 + ADR-0074 拆字段(2 ADR)
  → 推荐选 C 双指针(8d 迁移)
  → 风险:架构 commit 不可逆 + 双重维护陷阱
```

## 正例(2026-04-26 修正版)

```
Stage 5 发现三格式并存
  → 询问真实痛点(Beta 期 0 break)
  → 推迟 ADR-0072 到 M6+(状态 Deferred-to-M6)
  → ADR-0012 加 1 行 L0 immutability 澄清(无需新 ADR)
  → TASK-022 改 opt-in(无需强制迁移)
  → 节省 8d · 等 Exit-Beta 真实痛点出现再决策
```
